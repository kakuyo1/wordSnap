# wordSnapV1 未来规划（基于 TODOLIST 的可落地版本）

更新时间：2026-03-26

## 1. 规划输入与目标

本规划基于 `docs/TODOLIST.md` 的 12 条待办，目标是把“想做的事”转成“可执行、可验收、可发布”的里程碑计划。

核心目标：

- 在保持 V1 可用性的前提下，逐步补齐 AI 辅助、识别鲁棒性、测试体系、打包发布和稳定性治理。
- 每个阶段都必须有明确交付物、验收标准和回退方案，避免“只做功能不收口”。
- 把“长期项”（跨平台）拆成“短期可验证步骤”，先做可行性证明，再做大规模迁移。

## 2. TODOLIST 摘要（归并后）

原始 12 项需求可归并为 6 个主题：

1. **体验与交互**：结果卡片风格、结果状态规范、历史查询。
2. **智能增强**：接入 DeepSeek，输出精炼英文释义 + 词根词源。
3. **识别能力**：提升 OCR 效果、覆盖游戏/网页场景、处理全屏与热键冲突。
4. **工程质量**：TDD、单元/集成测试、性能与内存稳定性治理。
5. **交付能力**：`windeployqt + Inno Setup` 打包安装。
6. **平台演进**：开机自启、跨平台能力、头文件中文 Doxygen 注释规范。

## 3. 当前基线（可用但未收敛）

已具备：

- Qt6 + C++17 + CMake 工程闭环可运行。
- 托盘 + 全局热键 + 框选截图 + OCR + 词典查询 + 结果卡片展示链路。
- 设置项可落盘（热键、词典路径、tessdata、显示模式、卡片透明度）。

主要缺口：

- 仍无自动化测试（`ctest` 无测试用例）。
- AI 辅助能力尚未接入。
- OCR 在复杂背景/全屏游戏场景仍有不稳定风险。
- 打包安装、性能/内存治理流程尚未建立。

## 5. 详细执行计划（直到“实际可行”）

### M0：工程基线与测试骨架（必须先做）

目标：让后续需求都能“改得动、测得了、回得去”。

任务：

- 建立 `tests/` 目录，接入 CTest（建议先用 QtTest，减少依赖成本）。
- 为 `WordNormalizer`、`StarDictBackend`、音标提取逻辑补单元测试。
- 提供最小测试词典样本（小体积 `.ifo/.idx/.dict`）用于集成测试。
- 定义单测命名规范：`<Module>Test`，保证可 `ctest -R` 精准运行。
- 在 `README.md` 和 `AGENTS.md` 固化测试命令与执行顺序。

验收标准：

- `ctest --test-dir build -N` 可列出测试。
- 至少 8 个有效测试用例，覆盖正常/异常/边界输入。
- 单测命令可执行：
  - `ctest --test-dir build -R "^WordNormalizerTest$" --output-on-failure`

回退策略：

- 若测试框架引入阻塞构建，先保留核心测试 target，暂缓 UI 测试。

---

### M1：结果体验与状态规范

目标：统一用户看到的结果，减少“识别成功但不知道发生了什么”。

任务：

- 在设置中新增 ResultCard 风格选项（牛皮纸/玻璃拟态/终端/粘土）。
- 规范结果状态机与展示文案：
  - 成功命中（FOUND）
  - OCR 失败（OCR_FAILED）
  - 词典无结果（UNKNOWN）
  - 词典不可用（DICT_UNAVAILABLE）
- 新增“查询历史 V1”（本地持久化，建议 SQLite 或 JSONL）。
- 增加历史过滤（按词、按时间）和快速复查入口。

验收标准：

- 四类状态在卡片/托盘文案一致且可复现。
- 历史记录支持最近 N 条（建议默认 300）并可清空。
- 不影响现有热键抓取主链路。

---

### M2：AI 辅助解释（DeepSeek）

目标：在字典结果基础上提供“短、准、可控”的 AI 增强解释。

任务：

- 新增 `AiAssistService`，负责 API 检测、请求、重试、超时、降级。
- 软件启动时校验 API 配置：无效则自动禁用 AI，保留纯离线模式。
- 查询流程改为“两阶段渲染”：
  1) 先显示字典结果（词+音标+释义）。
  2) 异步请求 AI，底部显示加载动画（`...` 或点点动画）。
- 约束 AI 输出 JSON 结构：`definition_en`, `roots`, `etymology`。
- 输出限制：每项 `< 30` 英文词，失败时显示“AI 获取失败，不影响基础查词”。

验收标准：

- API 无效时不触发网络请求。
- 网络慢/失败不阻塞主线程、不影响基础结果展示。
- 至少 20 个词条抽样满足“结构正确 + 内容简练”。

---

### M3：OCR 鲁棒性与实战适配

目标：提高复杂背景、动态画面、全屏应用下的可用率。

任务：

- 预处理策略参数化（阈值、缩放倍率、反色尝试、多策略重试）。
- 增加 OCR 候选筛选策略（白名单字符、置信度阈值、候选投票）。
- 建立“实战样本集”（游戏、网页、深色/浅色背景、不同字号）。
- 处理全屏和热键冲突：
  - 提供备用热键建议；
  - 在冲突时给出可操作提示。
- 手工测试流程固化到文档（最少 30 个标准场景）。

验收标准：

- 样本集识别成功率达到阶段目标（建议 >= 85%）。
- 全屏场景下可稳定触发或给出明确失败反馈。

---

### M4：Windows 打包与安装

目标：把“开发机可跑”变成“用户机可安装可使用”。

任务：

- 提供一键打包脚本（构建 + `windeployqt` + 安装包生成）。
- 编写 Inno Setup 脚本（安装路径、开始菜单、卸载入口）。
- 安装后首启检查：词典目录、Tesseract 路径、设置初始化。
- 形成发布清单（版本号、变更日志、校验信息）。

验收标准：

- 全新 Windows 环境安装后可直接完成一次查词。
- 卸载后不残留主程序文件（用户数据可按策略保留）。

---

### M5：稳定性与性能收口

目标：让程序可长期后台运行而不明显劣化。

任务：

- 内存与句柄检查（Visual Studio Diagnostics / Dr.Memory 等）。
- 生命周期治理：对象释放、线程退出、异常路径清理。
- 引入性能指标采样：
  - 截图到首结果耗时；
  - OCR 耗时；
  - AI 耗时（若开启）。
- 实现开机自启配置（可开关）。

验收标准：

- 连续运行 2 小时内存曲线平稳、无持续增长异常。
- 崩溃可复现问题有 issue 与修复闭环。

---

### M6：跨平台可行性预研（Linux 优先）

目标：用最小成本判断是否进入跨平台实施期。

任务：

- 抽象平台层接口（热键、截图、托盘）并隔离 Win32 依赖。
- 在 Linux 上完成最小 PoC：托盘 + 截图 + OCR + 查词。
- 产出迁移差异清单（依赖、权限模型、桌面环境差异）。

验收标准：

- 给出明确的投入评估（人周、风险、收益）与建议结论。

## 7. 发布门禁（每个里程碑结束都要过）

- 能构建：`cmake --build build --config Debug`。
- 能运行：托盘可见、热键可触发、基本链路不回退。
- 能验证：已有测试必须通过；若新增核心逻辑，必须新增对应测试。
- 能回滚：关键行为变更要有降级开关或兼容策略。
- 能说明：`README.md`、`AGENTS.md`、`docs/` 文档同步更新。

## 8. 近期执行建议（下一个迭代）

建议下一迭代仅做 M0（不混入新功能），优先完成：

1. 建立测试框架与首批单测。
2. 把音标/释义解析逻辑中可测试部分抽离成纯函数模块。
3. 在文档中固化“单测优先于手工验证”的提交流程。

做到以上三点后，再进入 M1/M2，整体交付风险会明显下降。
