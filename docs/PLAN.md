# wordSnapV1 路线图与里程碑计划

更新时间：2026-03-27

## 1. 计划定位

- 本文档用于维护 `wordSnapV1` 的里程碑路线、状态与验收门槛。
- 规划以“可交付”为目标：每个阶段都必须有可验证产物、明确验收标准、可执行回退方案。
- 执行口径遵循 `docs/TDD_CORE.md`：先评估是否适合 TDD，核心逻辑优先 TDD，UI/样式类改动允许手工验证。
- 测试设计与新增测试范围需同时满足 `TESTING_GUIDELINES.md`（高价值、最小化、分层优先）。

- M0：测试基线已建立，CTest 与分层标签（`unit/integration/ui/e2e`）已接入。
- M1：结果卡片主题、状态规范、查询历史 V1 已落地。

- M5：长期运行稳定性与性能收口。
- M6：跨平台可行性预研。

## 4. 里程碑执行重点

### M3：OCR 鲁棒性与实战适配（本轮已收口）

目标：提高复杂背景、动态画面、全屏应用下的识别可用率与稳定反馈。

本轮结果（按当前需求）：

- 已完成预处理策略增强与相关单测回归。
- 已完成 M3-A 样本模板/评估模板。
- 已手工验证 `M3A-WEB-01~08`，结果为快速识别成功。

后续说明：

- M3 进一步优化（候选筛选、全屏冲突治理、全量基线补测）已转入 `docs/FUTURE.md`，待实际反馈触发再启用。

### M4：Windows 打包与发布流水线

目标：从“开发机可跑”升级为“目标用户可安装可用”。

关键任务：

- 一键打包脚本（构建 + `windeployqt` + 安装包生成）。
- Inno Setup 安装器脚本与安装后首启检查。
- 发布清单模板（版本号、变更摘要、校验信息）。

验收门槛：

- 新环境可一键安装并完成一次完整查词。
- 卸载行为符合预期（主程序清理，用户数据按策略保留）。

### M5：稳定性与性能收口

目标：支撑长时后台运行，避免性能与资源劣化。

关键任务：

- 内存/句柄基线与回归检测流程。
- 启动、截图、OCR、AI（可选）耗时指标采样。
- 生命周期与异常路径清理（对象释放、线程退出、错误分支）。

验收门槛：

- 连续运行 2 小时资源曲线稳定，无明显持续增长。
- 关键性能指标可量化、可追踪、可回归对比。

### M6：跨平台可行性预研（Linux 优先）

目标：以最小投入给出是否推进跨平台的明确结论。

关键任务：

- 抽象热键/截图/托盘平台边界，隔离 Win32 依赖。
- Linux 最小 PoC（托盘 + 截图 + OCR + 查词）。
- 迁移成本与风险评估报告（人周、依赖、权限模型、收益）。

验收门槛：

- 形成可执行的 Go/No-Go 决策结论。

## 5. 发布门禁（里程碑收口统一标准）

- 能构建：`cmake --build build --config Debug`。
- 能运行：托盘、热键、截图、OCR、查词主链路可用。
- 能验证：按改动类型执行对应验证（自动化测试或手工验证），并满足 `TESTING_GUIDELINES.md` 约束。
- 能回退：关键行为变更具备降级策略或兼容方案。
- 能同步：`README.md`、`docs/`、`AGENTS.md`（若受影响）同步更新。

## 6. 下一迭代建议（M4 优先）

1. 启动 M4：整理 Windows 一键打包脚本与安装器收口任务。
2. 保持现有 OCR 主链路稳定，收集真实使用反馈作为 `docs/FUTURE.md` 触发依据。
3. 若出现高频 OCR 失败场景，再按 `docs/FUTURE.md` 优先级恢复 M3 候选项。
