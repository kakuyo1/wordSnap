- 本文档用于维护 `wordSnapV1` 的里程碑路线、状态与验收门槛。

## 4. 里程碑执行重点

### M3：OCR 鲁棒性与实战适配（当前主线）

目标：提高复杂背景、动态画面、全屏应用下的识别可用率与稳定反馈。

执行策略（摘要）：

- 建立实战样本集与可复现实验流程，避免“只凭体感优化”。
- 在预处理、候选筛选、冲突处理三个层面分批收敛。
- 以“失败可解释”为底线：即使识别失败也要给出明确可操作提示。

验收门槛（阶段口径）：

- 样本集识别成功率达到阶段目标（建议 `>= 85%`，后续按样本分层再细化）。
- 全屏/冲突场景失败时，用户可收到明确原因与建议操作。

详细计划见：`docs/Temp/2026-03-27-M3-ocr-robustness-plan.md`。

### M4：Windows 打包与发布流水线

目标：从“开发机可跑”升级为“目标用户可安装可用”。

关键任务：

- 一键打包脚本（构建 + `windeployqt` + 安装包生成）。
- Inno Setup 安装器脚本与安装后首启检查。
- 发布清单模板（版本号、变更摘要、校验信息）。

验收门槛：

- 新环境可一键安装并完成一次完整查词。
- 卸载行为符合预期（主程序清理，用户数据按策略保留）。

### M5：稳定性与性能收口

目标：支撑长时后台运行，避免性能与资源劣化。

关键任务：

- 内存/句柄基线与回归检测流程。
- 启动、截图、OCR、AI（可选）耗时指标采样。
- 生命周期与异常路径清理（对象释放、线程退出、错误分支）。

验收门槛：

- 连续运行 2 小时资源曲线稳定，无明显持续增长。
- 关键性能指标可量化、可追踪、可回归对比。

### M6：跨平台可行性预研（Linux 优先）

目标：以最小投入给出是否推进跨平台的明确结论。

关键任务：

- 抽象热键/截图/托盘平台边界，隔离 Win32 依赖。
- Linux 最小 PoC（托盘 + 截图 + OCR + 查词）。
- 迁移成本与风险评估报告（人周、依赖、权限模型、收益）。

验收门槛：

- 形成可执行的 Go/No-Go 决策结论。

## 5. 发布门禁（里程碑收口统一标准）

- 能构建：`cmake --build build --config Debug`。
- 能运行：托盘、热键、截图、OCR、查词主链路可用。
- 能验证：按改动类型执行对应验证（自动化测试或手工验证），并记录结果。
- 能回退：关键行为变更具备降级策略或兼容方案。
- 能同步：`README.md`、`docs/`、`AGENTS.md`（若受影响）同步更新。

## 6. 下一迭代建议（M3 优先）

1. 先建立并冻结 M3 样本集与评分口径（作为后续优化基线）。
2. 再做预处理/候选筛选策略的分层实验与参数收敛。
3. 最后收口全屏冲突处理与用户提示文案，完成一次端到端回归。
