# wordSnapV1 TDD 核心原则（TDD_CORE）

更新时间：2026-03-27

> 本文件定义项目的 TDD 不变规则，原则上保持稳定；
> 若确需修改，必须在评审中说明原因，并同步更新 `docs/TDD_PLAN.md` 与 `AGENTS.md`。

## 1. 目标与适用范围

- 目标：通过“测试先行”降低回归风险，保证截图查词主链路可持续演进。
- 适用范围：`src/`、`tests/`、构建脚本与关键流程文档。
- 优先级：当交付速度与质量冲突时，优先保证可验证性与可回归性。

## 2. TDD 工作循环（必须遵循）

每个功能/修复按 `Red -> Green -> Refactor` 执行：

1. Red：先写测试，且测试应在当前代码下失败。
2. Green：以最小改动让测试通过。
3. Refactor：在测试保护下清理实现与重复逻辑。

执行要求：

- 不允许先改业务代码再“补测试”冒充 TDD。
- 每次提交至少包含一种可执行验证（自动化测试优先）。
- 修复缺陷必须先增加能复现该缺陷的测试。

## 3. 测试分层与职责

- 单元测试（unit）：验证纯逻辑/模块边界，依赖可控、执行快速。
- 集成测试（integration）：验证跨模块协作与关键编排链路。
- UI/行为测试（ui）：验证关键交互与边界行为（逐步引入）。
- 端到端烟雾（e2e）：覆盖最小可用主链路（少量高价值用例）。

原则：

- 以单元测试为主体；用集成测试覆盖主风险；谨慎增加 E2E 数量。
- 新增功能至少落在一个层级；跨模块改动必须补集成测试。

## 4. 用例设计准则

- 覆盖三类输入：正常、边界、异常。
- 覆盖三类路径：成功、失败、降级。
- 覆盖稳定契约：状态码、错误文案、超时与回退行为。
- 测试命名使用行为描述：`whenXxx_thenYyy` 或等价语义命名。

## 5. 完成定义（DoD）

任一需求完成前必须满足：

- 关联测试已新增/更新并通过。
- `cmake --build build --config Debug` 通过。
- `ctest --test-dir build -C Debug --output-on-failure` 通过。
- 文档已同步（至少更新 `docs/TDD_PLAN.md` 的进展与状态）。

## 6. 提交流程约束

- 默认小步提交：一次提交聚焦一个可验证改动。
- 提交信息应体现变更意图（如 `test:` / `fix:` / `refactor:`）。
- 禁止通过删除或弱化测试来“换取通过”。

## 7. 例外与补救

以下情况可临时偏离 TDD 顺序：

- 线上紧急故障需先止血（本项目当前以本地工具为主，通常不触发）。
- 第三方环境故障导致测试暂不可运行。

补救要求：

- 在同一迭代内补齐失败复现测试与回归测试。
- 在 `docs/TDD_PLAN.md` 记录偏离原因、补救动作和完成时间。
