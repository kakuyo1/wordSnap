# wordSnapV1 TDD 核心原则（TDD_CORE）

更新时间：2026-03-27

> 本文件定义项目 TDD 执行口径：**先评估、后决定是否 TDD**。
> 测试编写细则以 `TESTING_GUIDELINES.md` 为强约束，与本文件联合执行。
> 若修改本文件，需同步更新 `docs/TDD_PLAN.md` 与 `AGENTS.md`。

## 1. 目标与适用范围

- 目标：在可控成本下降低回归风险，优先保护高风险主链路与核心逻辑。
- 适用范围：`src/`、`tests/`、构建脚本与关键流程文档。
- 原则：TDD 是默认优先手段，但不是所有改动都强制“先写自动化测试”。

## 2. TDD 触发条件（先评估）

以下改动默认应执行 `Red -> Green -> Refactor`：

- 核心业务逻辑（OCR 结果判定、归一化、查词状态流转、降级策略）。
- 缺陷修复且可稳定复现（优先先写复现用例）。
- 跨模块协议/边界行为（错误码、超时、回退文案、参数组装）。
- 未来易回归且手工验证成本高的逻辑路径。

以下改动可不强制 TDD，改为“手工验证 + 必要 smoke”即可：

- UI 视觉样式微调（配色、字体、间距、阴影、动画细节）。
- 纯展示层文案/排版微调（不改变状态机与业务分支）。
- 一次性工程整理、低风险重命名、注释与文档修订。

## 3. 工作循环（按评估结果执行）

- 若判定“适合 TDD”：严格按 `Red -> Green -> Refactor`。
- 若判定“不适合 TDD”：
  - 在变更说明中写明“为何不做自动化测试”；
  - 给出可复现的手工验证步骤与预期结果；
  - 保留最小可运行检查（至少构建通过，必要时跑受影响 smoke）。

## 4. 测试分层策略（精简优先）

- 高价值优先：仅覆盖业务逻辑、公共 API、关键路径与高风险边界。
- 禁止 trivial 测试：不为 getter/setter、简单赋值、纯样式 glue 编写测试。
- 单元测试（unit）：主力层，优先覆盖纯逻辑与边界判定。
- 集成测试（integration）：用于高价值跨模块链路，避免泛滥。
- UI/行为测试（ui）：仅保留关键行为回归，不追求样式穷举。
- 端到端烟雾（e2e）：维持少量高价值用例，避免测试膨胀。
- 测试体量约束：单函数建议 3-5 个 case，优先参数化/数据驱动减少重复。
- 断言原则：只验证公开行为，不耦合实现细节。

## 5. 完成定义（DoD）

任一需求完成前满足以下其一：

- **TDD 路径**：新增/更新自动化测试并通过，且完成受影响面验证。
- **非 TDD 路径**：给出评估理由 + 手工验证记录，且完成受影响面验证。

通用要求：

- `cmake --build build --config Debug` 通过。
- 测试新增需符合 `TESTING_GUIDELINES.md`（价值、体量、分层、命名等约束）。
- 文档同步（至少更新 `docs/TDD_PLAN.md` 的进展与决策）。

## 6. 提交流程约束

- 默认小步提交：一次提交聚焦一个可验证改动。
- 提交信息应体现变更意图（如 `test:` / `fix:` / `refactor:` / `docs:`）。
- 禁止为了“看起来通过”而删除高价值测试。

## 7. 例外与补救

- 紧急止血、外部环境故障等场景可临时偏离既定流程。
- 偏离后需在同一迭代内补齐：
  - 能补自动化测试则补；
  - 不适合自动化测试则补完整手工回归记录。
- 所有偏离与补救都要在 `docs/TDD_PLAN.md` 留痕。
